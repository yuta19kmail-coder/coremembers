## 0. 結論サマリー（先に要点だけ）

1. **CoreMembers＝会社の正式な社員名簿＋組織図（人と組織の単一の真実）**。全スタッフ（ログインしない週2バイトも含む）を持つ**マスター**。
2. **portalMembers＝そのうち「アプリにログインできる人」のアクセス権レイヤー**に痩せさせる。CoreMembersの社員IDに**紐づくだけ**。
3. つまり【決定①】の答え＝**「完全別扱い」ではなく「親子リンク」**。CoreMembersが親（全社員）、portalMembersが子（アクセス権・部分集合）。1:1ではない（バイトはCoreMembersにだけ居る）。
4. **場所（陸運局A/B・部品商など）は `coreLocations` という設定で編集できるマスター**にして、人の「行ける場所」も各アプリの「行き先」もこれを参照する。
5. **運転スキル・免許・資格・適性**はCoreMembersの社員レコードに持ち、MHS/PitFlowのAI配車がここを読む。
6. **既存の本番portalMembersは壊さない**段階移行。apps は「人・組織＝CoreMembers」「ログイン可否＝portalMembers」に役割分担していく。

## 1. なぜ今CoreMembersなのか（背景・課題）

MHSを本番化して、担当（who）を `portalMembers` から引いて使い始めた。そこで次の要求が出てきた：

- 「担当に**田中**を選んだら、**2課**が自動で入ってほしい」
- 「**人ごと・課ごと**に予定を並べたい」
- 「**正社員だけ**スケジュールに出して、**週2バイトは出さない**」

ところが `portalMembers` は本来 **「このアカウントはどのアプリを開けるか・どの役割か」を決めるアクセス管理表**であって、

- ログインさせないバイトは **そもそも portalMembers に居ない**（＝担当に選べない／シフトに並べられない）
- 課/役職は CarFlow 用の `carflow.group` 等に断片的に入っているだけで、**会社の組織としての構造ではない**
- 雇用形態（正社員/パート/バイト）も上下関係（上長）も**どこにも無い**

→ **「人と組織の事実」を置く場所が無い**。これが根本原因。MHSの自動課・人別並べ・バイト除外も、PitFlowの配車（誰がMT可・どこへ行ける）も、全部この土台が無いと最終形にならない。

だから **CoreMembers ＝ 会社の正式な社員名簿＋組織図** を、人物Memory/AIより先に「土台」として作るのが正しい順番。

## 2. 役割の再定義（レイヤー分離）

| レイヤー | 何を持つ | 何に答える | 誰が編集 |
|---|---|---|---|
| **CoreMembers**（新・マスター） | 全社員＋組織図＋雇用＋スキル＋運転＋行ける場所＋人物Memory | 「**この会社に誰がいて、どういう組織で、何ができるか**」 | 人事（ゆうた・管理者） |
| **portalMembers**（既存・痩せる） | ログイン可否＋各アプリのロール（CarFlow管理者/StockFlow管理…） | 「**この人はどのアプリを開けて、何ができるか**」 | 管理者（ゆうた） |
| **coreLocations**（新・場所マスター） | 陸運局A/B・部品商・拠点・顧客先 等 | 「**行き先・方面の定義**」 | 設定（管理者） |

ポイント：**「人・組織」と「アクセス」は別の関心事**。混ぜると今のように破綻する。CoreMembersは人事の真実、portalMembersは入退室の鍵。

## 3. 【決定事項①】CoreMembers と portalMembers の関係

### 答え：完全別扱いではなく「親子リンク（CoreMembers が親）」

- **CoreMembers＝全社員（スーパーセット）**。1人1レコード。ログインしないバイトもここに居る。
- **portalMembers＝アクセス権（サブセット）**。ログインできる人にだけ存在し、**`memberId` で CoreMembers の社員を指す**。
- だから関係は **1:1 ではなく「CoreMembers : portalMembers ＝ 1 : 0または1」**。
  - 例）社長＝CoreMembersにいる＋portalMembersにもいる（ログインする）。
  - 例）週2バイトの佐藤＝CoreMembersにだけいる（ログインしない・でもシフトや配車には登場しうる）。

### なぜ「完全別扱い」にしないか
別々に二重管理すると、名前変更・退職・写真が両方に出てズレる（CarFlow/StockFlow初期で起きた事故と同じ）。**真実は1か所（CoreMembers）**にして、portalMembersはそこへの参照＋鍵情報だけにするのが破綻しない。

### なぜ「1:1 統合（portalMembersを廃止）」にもしないか
- portalMembersは**本番3アプリ（CarFlow/StockFlow/CoreFlow）が今この瞬間に入室判定で読んでいる**。いきなり廃止は危険。
- 「アクセス権」は人事情報とは更新頻度も責任者も違う（セキュリティ）。レイヤーを分けたまま**リンクで結ぶ**のが安全。

### 移行の方向（詳細は§9）
1. CoreMembers を作り、**現portalMembersの全員を社員レコードとして取り込み（seed）**＋ `portalMemberId` で相互リンク。
2. その後 **ログインしないバイト等を CoreMembers にだけ追加**。
3. apps は徐々に「人・組織の読み取り先」を CoreMembers に切替（portalMembersはアクセス判定専任に）。

## 4. データモデル（Firestore・carflow-9d500 相乗り）


## 5. 場所マスター `coreLocations`（設定で変更したい件）

- **陸運局A/B・部品商・自社拠点・顧客先**などを **1か所のマスター**にして、設定画面（CoreMembers内 or CoreFlow）で **追加/編集/並べ替え/無効化**できる（MHSの「会社カレンダー設定」と同じ作法）。
- 各社員の **「行ける場所」** はこのマスターを**参照（ID）**するだけ（文字列でバラ持ちしない＝表記ゆれ防止）。
- **`area`（方面）** を持たせるのが配車の肝：「同じ柏方面をまとめて取る」のAI素材になる。
- MHSの **場所軸ビュー**、PitFlowの **配車の行き先**、両方がこれを唯一の真実として使う。
- 将来 `lat/lng` を入れれば距離・ルート最適化（ついで取り）の精度が上がる（最初は方面の手動分類で十分）。

## 6. 各アプリがどう使うか（連携設計）

| アプリ | CoreMembers から読むもの | 効果 |
|---|---|---|
| **MHS** | 社員（showInSchedule=true・在籍）／primaryDeptId／coreLocations | 担当に人を選ぶと**課が自動**／**人別・課別**で並べる／**バイト除外**／場所軸＝coreLocations |
| **PitFlow** | 運転スキル(MT/サイズ)・goableLocations・area・整備資格 | **AI配車**（誰がどの陸運局へ・MT可・ついで部品商）／作業割当 |
| **CoreFlow** | 社員＋組織（人事レイヤー）＋portalMembers（アクセスレイヤー） | メンバー管理を**2層UI**化（人事＝CoreMembers／鍵＝portalMembers） |
| **CarFlow / StockFlow** | 当面は今まで通り portalMembers（アクセス）。表示名/写真は将来 CoreMembers に寄せられる | 既存運用は不変。徐々に表示を CoreMembers 由来へ |

**入室判定（ログイン）は引き続き portalMembers**。CoreMembers は「人と組織」専任。

## 7. 権限・プライバシー設計（人物Memoryの本丸・最重要）

人物Memory は人事ハラスメント／プライバシー紙一重。**実装前に運用ルール1枚を全員で握る**（暫定仕様書§プライバシー参照）。要点：

- **閲覧**：本人＝自分の全部／**上長(managerId)**＝部下の中身／経営者＝全員／同僚＝話題タイトルのみ。
- **視点別生成**（self / peer / supervisor）で「クローズ過ぎずオープン過ぎず」を実現。
- 「上司が見る前提」を本人に明示（録音同意）。事実と主観の分離、本人の修正/削除権、退職者データの扱い。
- Firestoreルールでも **coreMemories は organizational に厳格**（本人＋上長＋経営者のみ。社員プロフィールの一般項目とは別ルール）。

> ※「名簿＋組織図＋スキル＋場所」（§4の上半分）は社内共有してよい情報。**人物Memory（coreMemories/corePatterns）だけ別格でセンシティブ**として権限を分ける。

## 8. 権限ロール（CoreMembers内）

| ロール | 名簿/組織/スキル | 人物Memory | 編集 |

| バイト等（非ログイン） | — （本人は使わない・名簿に載るだけ） | — | — |

## 9. 移行計画（本番portalMembersを壊さない）

```
Step 1  CoreMembers コレクション新設＋seed
        ・現 portalMembers 全員を coreMembers に取り込み（name/photo/role/dept断片を移植）
        ・相互リンク：coreMembers.portalMemberId ↔ portalMembers.memberId
        ・この時点で apps は不変（まだ portalMembers を読んでいる）

Step 2  組織図と場所マスターを整える
        ・coreDepts に 部/課（整備部>1課/2課・受付・車販・デザイン 等）を入力
        ・coreLocations に 陸運局A/B・部品商・拠点 を入力
        ・各社員に primaryDeptId / goableLocationIds / 雇用形態 を付与

Step 3  非ログインスタッフを追加
        ・週2バイト等を coreMembers にだけ登録（portalMembers無し）

Step 4  apps を順に切替（人・組織の読み先を CoreMembers へ）
        ・MHS：担当ピッカー＝coreMembers（自動課・バイト除外）／場所軸＝coreLocations
        ・PitFlow：配車＝coreMembers運転スキル＋coreLocations
        ・CoreFlow：メンバー管理を「人事＋アクセス」の2層に

Step 5  portalMembers を痩せさせる（任意・最後）
        ・重複プロフィール項目を CoreMembers 参照に寄せ、portalMembers は access+role 専任へ
        ・※急がない。Step4までで実用上はほぼ達成
```

⚠ 各 Step は **追加のみ**で進め、既存の入室判定（portalMembers）には触らない。CarFlow/StockFlow/CoreFlow/MHS が止まらないことを最優先。

## 10. 開発フェーズ

```
Phase 0  モック（動かせるHTML1枚）
         ・名簿一覧／社員カード（組織・雇用・運転・場所）／組織図／場所マスター設定 のラフ

Phase 1  土台＝名簿＋組織図＋場所マスター（★最優先・MHS/PitFlowが欲しいのはここ）
         ・coreMembers / coreDepts / coreLocations の CRUD
         ・portalMembers から seed＋相互リンク
         ・雇用形態・showInSchedule・運転スキル・行ける場所の入力UI

Phase 2  MHS / PitFlow 連携
         ・MHS：担当=coreMembers（自動課・人別/課別・バイト除外）、場所軸=coreLocations
         ・PitFlow：運転スキル＋場所を配車素材として供給

Phase 3  スキルマップ・資格管理の作り込み
         ・整備士資格/免許の一覧・マップ、誰が何できる俯瞰

Phase 4  人物Memory（手動入力・本人/上司ビュー）  ← 暫定仕様書の本丸その1
         ・要約テキストの手入力/取り込み、人物軸の蓄積、権限分離

Phase 5  AI集約・パターン分析・配車AI連携  ← 本丸その2
         ・Claude APIで視点別生成・パターン抽出
         ・MHS/PitFlowのAI配車提案に運転スキル＋場所＋負荷を供給
```

**「次の正式開発」で着手するのは Phase 1（土台）まで**を想定。人物Memory/AI（Phase4-5）はその後。土台ができれば MHS/PitFlow の「自動課・人別並べ・バイト除外・配車」が一気に正しくなる。

## 11. 未決事項・次セッションで握ること

- [ ] **組織図の深さ**：部>課 の2階層で足りるか（班・拠点・兼務・上長まで要るか）。当面MHS/PitFlowが使うのは「課・雇用・在籍・上長」程度の見込み。
- [ ] **雇用区分の正式リスト**（役員/正社員/契約/パート/アルバイト/外部委託 …）と、各区分の既定 `showInSchedule`。
- [ ] **運転スキルの項目粒度**（免許種別の選択肢・MT可否・サイズ・特殊車両・得意度の尺度）。
- [ ] **場所マスターのカテゴリと方面（area）の定義**（柏方面/市川方面…の括り方）。
- [ ] **portalMembers との同期方向**（CoreFlowでの編集UIを CoreMembers 側に寄せるか、当面両方残すか）。
- [ ] **人物Memoryの運用ルール1枚**（実装前に必須・プライバシー）。
- [ ] **アプリの色/絵文字**（橙#ea580c系・👥）。
- [ ] CoreMembers を **CoreFlowのメンバー管理に統合**するか、**独立アプリ**にするか（人事＝独立／アクセス＝CoreFlow、という分け方が自然そう）。
