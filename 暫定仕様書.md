## 💡 なぜ作るか（背景・課題）

ゆうたは現在、ジェンスパークのAI会議機能で MTG・1on1・指導内容を録音→要約している。
**しかしその要約は記録として散らばっていて、データとして活用できていない**：

- 直近の振り返りには使えるが、人物軸での蓄積になっていない
- 「Aに3ヶ月前と同じ指摘をした」が見えない
- 「Aは複数人から同じ指摘を受けてる→個人特性かも」が見えない
- 上司側からは「誰がどう伸びてる/詰まってるか」の俯瞰ができない
- 本人側からも「自分の長期的課題は何か」が見えない

つまり **「人事に関する記憶を、ゆうたの頭の中ではなく見える場所に置きたい」** が原動力。

加えて、メンバー登録（CoreFlow `portalMembers`）は既に必要。であれば、**プロフィール＋スキル＋運転可否＋行ける場所＋適性 まで含めた「拡張社員名簿」**にしてしまうのが効率的。ScheduleFlow の AI配車提案にも必要になるデータ供給源にもなる。

## 🧩 主要機能

### A. 拡張社員名簿（プロフィールタブ）
- 基本情報（CoreFlow `portalMembers` から自動引き継ぎ・名前/写真/役職/部署/uid）
- 在籍情報・雇用形態
- スキル・資格・免許（整備関連の有資格・運転免許の種類）
- **運転できる車**：MT/AT・サイズ・特殊車両 ★ScheduleFlowが読む
- **行ける場所**：陸運局A/B、部品屋、顧客先 ★ScheduleFlowが読む
- 専門領域・得意/苦手のタグ
- 適性検査データ（将来・任意）

### B. 人物Memory（本丸）
**録音→要約されたMTG・1on1・指導の内容を、人物軸で永続的に蓄積。**
- 1on1ログ
- MTGログ
- 指導ログ（誰が誰に何を）
- 評価面談ログ
- 自由メモ（ゆうたの観察等）

入力経路：
- ジェンスパークの要約をコピペ or ファイル取り込み（暫定）
- 将来：録音ファイルアップロード → Claude が文字起こし → 要約 → 振り分け（全自動）

### C. AI集約・パターン分析（Claude API）
入力された要約から自動で：
1. **登場人物の特定**（誰のことか・誰が話してるか・CoreFlow名簿と照合）
2. **テーマ分類**（接客／技術／態度／勤怠／成長／etc）
3. **アクションアイテム抽出**（誰がいつまでに何を）
4. **パターン検出**：
   - 繰り返し指摘されてるテーマ
   - 複数人から同じ指摘 → 個人特性の可能性
   - 改善トラッキング（前回指摘との照合）
   - 指導関係マップ（誰が誰を育ててる）
   - 退職リスク指標（ネガティブ偏向・改善停滞）

### D. 視点別ビュー（プライバシー設計の本丸）
**同じ1件のログから、Claude が「本人用」「同僚用」「上司用」の3バージョンを並列生成。**
クローズ過ぎず、オープン過ぎない中間運用を実現。

#### 🟢 本人ビュー
- 今日のテーマ・指摘内容
- 過去の関連（同テーマの履歴）
- 自分のアクション・期限
- 長期テーマ・成長目標

#### 🟡 同僚ビュー（限定的）
- 「何の話題か」だけ表示（中身は本人と上司のみ）
- 「最近 接客 について話し合った」レベル
- プライド保護と透明性の両立

#### 🔴 上司ビュー
- 具体的指摘内容
- パターン分析の結果
- 配置示唆・育成提案
- リスクフラグ

### E. 本人ダッシュボード
- 自分の課題マップ
- 受けたアドバイス履歴
- 改善トラッキング（グラフ）
- 次のアクション

### F. マネージャーダッシュボード
- 全員の俯瞰マップ
- 指導関係の可視化
- スキルマップ（誰が何強い・弱い）
- 育成カレンダー・後継者候補
- リスクアラート（要観察スタッフ）

## 🛡️ プライバシー・権限設計（重要・運用ルール）

**「人物Memoryアプリ」は人事ハラスメント or プライバシー侵害紙一重。設計段階で握っておかないと事故る。**

最低限決めとくべき項目（運用ルールとして別ドキュメント化予定）：

1. **誰が誰のログを見れるか**
   - 本人：自分のは全部見える
   - 直属の上司：部下のは見える
   - 経営者（ゆうた・社長・専務）：全員見える
   - 同僚：話題タイトルレベルだけ見える（中身は隠す）
2. **「上司が見てる前提」を本人にも明示**（録音時の同意取得）
3. **書き込み権限**（誰が誰のドキュメントに書ける？）
4. **「ネガティブ評価」の扱い**（事実 vs 主観の分離・本人の反論機会）
5. **退職者データの扱い**（即削除？アーカイブ？採用判断に再利用？）
6. **本人の修正/削除権**（個人情報保護法上の要請）

→ **GrowFlow 本実装前に「運用ルール 1枚」を策定して全員で握る。**

## 🗂️ データ構造（ラフ案）

```
companies/kobayashi_motors/
├ growProfiles/{uid}                    ← 拡張社員名簿（CoreFlow portalMembers と1:1）
│   ├ basic（CoreFlowから複製）
│   ├ skills, licenses, drivableCars
│   ├ goablePlaces
│   ├ traits（特性タグ）
│   └ assessments（適性検査・将来）
│
├ growMemories/{memoryId}               ← 1件のMTG/1on1/指導記録
│   ├ source: '1on1'|'mtg'|'instruction'|'review'|'memo'
│   ├ date, recordedBy
│   ├ rawText（要約原文）
│   ├ participants: [uid, uid, ...]
│   ├ aboutPerson: uid（誰についての話か）
│   ├ themes: ['接客', '技術']
│   ├ actions: [{forUid, what, dueDate}]
│   └ views（Claudeが生成）
│       ├ self（本人向け要約）
│       ├ peer（同僚向け要約・話題のみ）
│       └ supervisor（上司向け要約・分析つき）
│
└ growPatterns/{uid}                    ← Claudeが定期更新するパターン分析
    ├ repeatedThemes
    ├ multiPointed（複数人からの同種指摘）
    ├ improvementTracking
    └ riskFlags
```

## 📅 開発段階

```
Phase 0 ─ モック（並行作業）
  ・HTML1枚で雰囲気共有
  ・主要画面のラフ（プロフィール／Memory入力／本人ダッシュ／上司ダッシュ）
  
Phase 1 ─ 拡張社員名簿だけ実装
  ・スキル・運転・行ける場所の入力UI
  ・ScheduleFlow への供給開始
  ・CoreFlow portalMembers との同期
  
Phase 2 ─ 人物Memory 入力＆閲覧
  ・要約テキストの手動入力 or ファイルアップロード
  ・人物軸の蓄積（まだAI集約なし）
  ・本人/上司の閲覧UI
  
Phase 3 ─ AI集約・パターン分析（本丸）
  ・Claude API で振り分け・視点別生成・パターン抽出
  ・本人/同僚/上司の3視点ビュー
  ・繰り返しテーマ・指導関係マップ
  
Phase 4 ─ 録音→要約まで自前化（将来）
  ・録音ファイルアップロード → Claude が文字起こし → 要約 → 振り分け
  ・ジェンスパークから乗り換え
```
